# Contributing to miolimOS

Thanks for your interest in miolimOS! This project has a deliberately
**maintainer-driven contribution model**, so please read this before
opening a pull request.

## How miolimOS is built (please read)

miolimOS is developed and maintained primarily by its owner together with
an automated build agent. To keep the codebase coherent, secure and
sustainable, **the preferred way to contribute is not a code pull request,
but a well-described feature request or "user story"** that the maintainer
then implements.

This is the same model successful projects like **SQLite** use: it keeps
quality and direction under one consistent hand while still being fully
open for everyone to use and self-host. The benefit for you: you don't
have to match an unfamiliar codebase's conventions — you describe the
*outcome* you want, and it gets built properly.

### So, in order of preference

1. **Open an issue describing the user story / problem / desired
   outcome.** What are you trying to achieve, and why? This is the most
   useful and most welcome form of contribution. Most accepted changes
   start here and are implemented by the maintainer.
2. **Bug reports** with clear reproduction steps are very welcome as
   issues.
3. **Pull requests are accepted only at the maintainer's discretion.**
   Small, obviously-correct fixes (typos, docs, clear one-line bugfixes)
   are the most likely to be merged. For anything larger, please open an
   issue first to agree on the approach — a PR written without that risks
   being declined regardless of its quality.

## If you do submit a pull request

- You will be asked to sign the **[Contributor License Agreement](CLA.md)**
  (handled automatically by a bot on your first PR). This lets the project
  stay dual-licensable while **you keep the copyright to your work**. See
  CLA.md for the plain-language rationale.
- Keep PRs focused and small. One logical change per PR.
- Match the surrounding code style; run the test suite.

## Security

Please do **not** open public issues for security vulnerabilities. Report
them privately to the maintainer instead.
